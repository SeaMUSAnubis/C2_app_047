### 2. Backend raw log collection

Them lop raw log rieng, khong tron truc tiep voi `event_logs` normalized hien co.

Them bang PostgreSQL `raw_user_logs`:

- `id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `source_id TEXT NOT NULL UNIQUE`
- `collector_type TEXT NOT NULL`
- `event_type TEXT NOT NULL`
- `timestamp TEXT NOT NULL`
- `user_id TEXT`
- `device_id TEXT`
- `raw_payload_json TEXT NOT NULL`
- `ingest_metadata_json TEXT NOT NULL DEFAULT '{}'`
- `normalized_event_id INTEGER REFERENCES event_logs(id)`
- `created_at TEXT NOT NULL`

Them indexes:

- `idx_raw_user_logs_timestamp`
- `idx_raw_user_logs_user`
- `idx_raw_user_logs_device`
- `idx_raw_user_logs_event_type`

Upsert behavior:

- Unique by `source_id`.
- Repeated ingest updates raw payload and metadata, preserving row id.

Supported `event_type` values align voi CERT-style sources:

- `logon`
- `device`
- `file`
- `http`
- `email`
- `process`
- `network`
- `ldap`
- `psychometric`
- `custom`

`ldap` va `psychometric` duoc chap nhan nhu raw enrichment data, nhung khong dung nhu primary behavioral event logs tru khi normalization sau nay map chung sang event.

### 3. API/interface changes

Them Pydantic schemas:

- `RawLogIngest`
- `RawLogRead`
- `RawLogBatchIngest`
- `RawLogBatchResult`

Them endpoints:

- `POST /api/raw-logs/ingest`
  - Requires authenticated admin or analyst.
  - Accepts one raw log record.
  - Returns stored raw log.
- `POST /api/raw-logs/batch`
  - Requires authenticated admin or analyst.
  - Accepts up to `1000` records per request.
  - Returns `{created_or_updated, failed, errors}`.
- `GET /api/raw-logs`
  - Requires authenticated admin or analyst.
  - Filters: `user_id`, `device_id`, `event_type`, `collector_type`, `limit`, `offset`.
- `GET /api/raw-logs/{id}`
  - Requires authenticated admin or analyst.
  - Returns one raw record.

Request shape for single ingest:

```json
{
  "source_id": "agent:PC-1001:logon:2026-06-15T08:15:00Z",
  "collector_type": "endpoint_agent",
  "event_type": "logon",
  "timestamp": "2026-06-15T08:15:00Z",
  "user_id": "ACM0001",
  "device_id": "PC-1001",
  "raw_payload": {
    "action": "Logon",
    "username": "acm0001",
    "pc": "PC-1001"
  },
  "ingest_metadata": {
    "agent_version": "0.1.0",
    "host_os": "Windows 11"
  }
}
```

Response shape mirrors request plus:

- `id`
- `normalized_event_id`
- `created_at`

### 4. Data flow

Raw ingest flow:

1. Agent/frontend/importer sends raw user/device activity to `/api/raw-logs/ingest` or `/api/raw-logs/batch`.
2. Backend stores raw JSON in `raw_user_logs`.
3. Existing normalized `event_logs` remains the canonical contract for dashboard, timeline and ML.
4. Later normalization job maps raw logs into `event_logs`.
5. Feature builder reads normalized events, not raw logs, unless a later task explicitly adds raw-to-feature direct processing.

This keeps raw evidence auditable while preventing ML/dashboard code from depending on source-specific payload shapes.

## Test Plan

Add unit tests:

- Raw log schema validates required fields and preserves arbitrary `raw_payload`.
- Raw log service uses PostgreSQL `%s` placeholders.
- Single ingest uses `ON CONFLICT(source_id) DO UPDATE`.
- Batch ingest returns per-record errors without failing the whole batch.
- JSON fields decode from `raw_payload_json` and `ingest_metadata_json`.
- Filters generate expected `WHERE` clauses for user/device/event type/collector.

Add API tests:

- Unauthorized raw log ingest returns auth error.
- Admin can ingest one raw log.
- Analyst can ingest one raw log.
- Duplicate `source_id` is idempotent and returns the same row id.
- Batch ingest stores valid rows and reports invalid rows.
- `GET /api/raw-logs` filters by `event_type`.
- `GET /api/raw-logs/{id}` returns 404 for missing row.

## Assumptions

- Runtime database remains PostgreSQL only; no SQLite fallback.
- Existing `POST /api/logs/ingest` remains for normalized events.
- New raw collection endpoints use `/api/raw-logs/*` to clearly separate raw data from normalized logs.
- `raw_user_logs` stores source-specific payloads as JSON text for now; no Postgres JSONB migration unless explicitly requested.
- PII/secrets are not filtered in v1; caller is responsible for sending only security metadata needed for UEBA. A later privacy task should add redaction policy.
